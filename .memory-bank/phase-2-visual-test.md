## 🎯 Что проверяем и зачем

Phase 2 — рефакторинг рендера сообщений: вместо «все 4000+ карточек в DOM» теперь только видимые ~20 строк + запас (`react-window` 2.2). Все 176 vitest + 37 cjs тестов и CI зелёные. Но **код этого уровня нельзя проверить статически** — нужны живые сценарии:

- Лента в больших чатах должна крутиться плавно (главная цель Phase 2)
- Открытие чата на «Новые сообщения» через `scrollToRow` вместо `scrollIntoView`
- Reply-клик находит сообщение **вне** видимого DOM через `findRenderItemIndex`
- Mark-read через `IntersectionObserver` с `root=listRef.element` (а не `null`)
- Подгрузка вверх/вниз не «дёргает» позицию (защищена `overflow-anchor: none`)

## 📝 Журнал найденных проблем и исправлений

Заполняется по ходу проверки. Каждая запись — одна найденная проблема + что было сделано.

### Проблема #1 — 14 мая 2026

**Пункт чек-листа**: косвенно касается **1, 3** (визуальная корректность ленты).

**Симптом** (по скриншотам пользователя):
- Time-divider («17:30») и имя отправителя следующей группы («Автовоз») визуально склеены — расположены практически на одной горизонтальной линии вместо нормального вертикального отступа между ними.
- При прокрутке вниз в чате «Автовоз» (~5000 сообщений) окно «дёргается вверх-вниз» — лента откатывается обратно на 100-200 пикселей в момент скролла.

**Что показали логи** (`%APPDATA%/ЦентрЧатов/chatcenter.log`):
- `scrollHeight` меняется с `73129` до `74064` за секунду — react-window асинхронно пересчитывает позиции после ResizeObserver измерил реальные высоты row.
- В момент скролла видно откат `top=8831 → 8733` (на 98 px назад) — это и есть «дёрг».

**Причина** (две связанные):
1. CSS-классы `.native-msg-divider`, `.native-msg-day-row`, `.native-msg-unread-divider` использовали `margin: 14px 0 6px` для отступов между разделителями. React-window 2.x измеряет row через `ResizeObserver` в border-box режиме — **margin не входит в измеренную высоту**. Следующий row начинается сразу после `height` предыдущего, наезжая на margin-space → визуальное склеивание дивайдеров с именем отправителя.
2. `align-self: center` в `.native-msg-divider` работал только внутри прежнего flex-column scroll-container'a. В react-window row — `position: absolute` без flex-родителя → центрирование сломалось.
3. `defaultRowHeight: 70` слишком велик для реальных размеров (дивайдеры ~25-30 px, короткие сообщения ~40-60 px). Большая дельта «до measure / после measure» = заметный откат scrollTop.

**Исправление** ([src/native/components/VirtualMessageList.jsx](src/native/components/VirtualMessageList.jsx)):
- `MessageRow` для каждого типа row задаёт **inline `paddingTop`/`paddingBottom`** вместо опоры на CSS margin → отступы попадают в border-box → ResizeObserver учитывает их → следующий row начинается с правильного смещения.
- На самих дивайдер-span'ах **обнулил `margin: 0`** inline чтобы не было двойного отступа с CSS-классом.
- На wrapper-row добавил `display: flex; justifyContent: center` (для `time`) и `width: '100%'` (для всех) → горизонтальное центрирование работает.
- Уменьшил `defaultRowHeight` **70 → 50** → меньше дельта при первом измерении → меньше «дёргание» scrollTop.

**Коммит**: `9079084 fix(virt): дивайдеры криво + дёргание scrollTop при скролле (Phase 2 проблема #1)`

**Статус**: 🟡 Исправление запушено. Ждёт повторной визуальной проверки пользователем.

### Проблема #2 — 14 мая 2026 — НЕ ОТ Phase 2 (давний баг)

**Симптом** (по скриншоту пользователя):
- Чат «Дугин Алексей Сергеевич» (личный, через аккаунт Avtoliberty) при открытии показывает **только одно сообщение** «111 (12:57)», хотя в реальности там была долгая переписка.
- Пользователь сообщает: «не всегда грузится вся история» — иногда история есть, иногда нет.

**Что показали логи** (`%APPDATA%/ЦентрЧатов/chatcenter.log`):

```
12:57:59  push «111» от Telegram → messages[dugin] = [{id:537823}]
12:58:03  chat-open chatId=tg_638454350:611696632
          messages=1  firstId=537823  lastId=537823  readInboxMaxId=537787
```

Признак: `readInboxMaxId=537787` < `firstId=537823` (разрыв 36 id) и **нет вызова `get-messages`** для этого chatId. То есть приложение не пыталось скачать историю с сервера.

**Причина** ([src/native/modes/InboxMode.jsx:66](src/native/modes/InboxMode.jsx#L66)):

```jsx
if (!store.messages[store.activeChatId]) {
  store.loadMessages(store.activeChatId, 50)
}
```

Загрузка истории запускается **только если массив `messages[chatId]` пустой**. Если push-уведомление пришло ДО открытия чата — кэш уже заполнен одним элементом → условие `false` → история не запрашивается.

**Воспроизведение**:
1. Чат закрыт (или приложение перезапущено).
2. Telegram push доставил **новое** сообщение в этот чат.
3. Пользователь открывает чат → видит только это одно сообщение.

**Связь с Phase 2**: ❌ Нет. Phase 2 трогал только рендер (`InboxChatPanel`, `VirtualMessageList`). Загрузка истории (`useEffect` в `InboxMode` + `nativeStore.loadMessages`) не менялась. **Этот баг был и до v0.89.0.**

**Возможное исправление** (когда понадобится — отдельным коммитом, **сейчас НЕ правим**):

```jsx
const cached = store.messages[store.activeChatId] || []
const firstCached = cached[0]?.id
const needHistory = cached.length < 20
  || (chat?.readInboxMaxId && firstCached && Number(firstCached) > Number(chat.readInboxMaxId))
if (needHistory) store.loadMessages(store.activeChatId, 50)
```

Логика: грузим историю если кэш тонкий (< 20 сообщений) **или** если в кэше нет сообщений старее `readInboxMaxId` (т.е. между прочитанным курсором и кэшем есть разрыв).

**Статус**: 📋 Зафиксировано. Пользователь попросил **сейчас не править**, только понять ситуацию.
