## Зачем это делаем

Сейчас в интерфейсе уже есть точки:

- на верхних WebView-вкладках;
- на аватарках аккаунтов слева в native `ЦентрЧатов`.

Но они означают разные вещи:

- верхняя точка сейчас показывает `monitorStatus` — состояние внутреннего монитора уведомлений;
- левая точка native-аккаунта сейчас показывает `account.status === 'connected'` — Telegram API account подключён;
- ни одна из этих точек прямо не отвечает на пользовательский вопрос: **интернет/сайт/аккаунт нормально отвечает или тормозит?**

Из-за этого можно сделать ложный вывод: например, слабый интернет выглядит как баг `start:prodlike`, VK/MAX или приложения.

Новая цель: точка всегда значит одно и то же:

```text
качество подключения / доступность
```

## Новая пользовательская модель

### Значения точки

```text
⚪ ожидание проверки
🟢 работает
🟡 медленно
🔴 не отвечает / ошибка
```

Смысл:

- `⚪ ожидание проверки` — вкладка/аккаунт уже есть в UI, но первая проверка ещё не завершилась;
- `🟢 работает` — последнее подключение/загрузка/ответ прошли быстро;
- `🟡 медленно` — подключение есть, но ответ долгий, например 10-15+ секунд;
- `🔴 не отвечает / ошибка` — timeout, `did-fail-load`, долгий `Not connected`, crash или другая подтверждённая ошибка.

### Главное правило

```text
Точка не показывает monitorStatus.
Точка показывает только качество подключения.
```

Внутренний мониторинг уведомлений остаётся, но его статус больше не является главным UI-сигналом.

## Где показываем

### Верхние WebView-вкладки

```text
Telegram 🟢    ВКонтакте 🟡    WhatsApp 🟢    Telega ⚪    Макс 🔴
БНК            Рабочий                         Avtoliberty
```

Для WebView статус берётся из загрузки и ошибок страницы:

- `did-start-loading` → `pending`;
- быстрый `dom-ready` / `did-finish-load` → `ok`;
- загрузка дольше порога → `slow`;
- `did-fail-load`, crash, timeout → `error`;
- повторная успешная загрузка → снова `ok`.

### Native/API аккаунты слева

```text
[БНК 🟢]
[Avtoliberty 🟡]
[VK Рабочий 🔴]       будущие аккаунты
[MAX ⚪]              будущие аккаунты
```

Для native/API аккаунтов статус берётся из IPC/API-операций:

- account создан, но первая проверка не прошла → `pending`;
- `tg:get-accounts`, `tg:get-cached-chats`, `tg:get-chats` быстро ответили → `ok`;
- ответ долгий или были reconnect/timeouts, но всё восстановилось → `slow`;
- аккаунт отключён, ошибка сессии, долгий `Not connected` без восстановления → `error`.

## Найденные проблемы после ручной проверки 8 мая 2026

Статус: требуется следующий этап исправления. Текущая реализация полезна как основа, но в ней обнаружены две ошибки в смысле показанных цифр.

### Проблема 1 — WebView показывает ложные секунды

Симптом:

```text
ВКонтакте — Работает — 392 сек
Telegram — Работает — 238 сек
Макс — Работает — 19 сек
```

Почему это неправильно:

- зелёный статус `Работает` означает, что вкладка живая;
- но число справа сейчас может быть не временем ответа страницы, а временем от старого `startedAt`;
- из-за этого пользователь видит зелёный статус и огромные секунды, хотя это не обязательно реальная скорость текущего ответа.

Причина в коде:

- `src/utils/consoleMessageHandler.js` получает полезные `__CC_*` сигналы от WebView и вызывает `markHealthOk`;
- `src/utils/connectionHealth.js` в `markHealthOk` при отсутствии явного `lastMs` может брать старый `prev.startedAt`;
- если `startedAt` старый, `lastMs` превращается в “сколько прошло с давнего события”, а не “за сколько ответила вкладка”.

Вывод:

```text
Для WebView нельзя использовать одно поле lastMs для разных смыслов.
```

Нужно разделить минимум два значения:

- время загрузки страницы (`lastLoadMs`);
- факт живого ответа/последней проверки (`lastSignalAt`, `lastCheckedAt`).

### Проблема 2 — Telegram API аккаунты получают общий замер вместо личного

Симптом:

```text
telegram · БНК        Работает / Медленно отвечает   2.9 сек / 12 сек
telegram · Avtoliberty Работает / Медленно отвечает  2.9 сек / 12 сек
```

Почему это неправильно:

- в программе два отдельных Telegram API аккаунта;
- каждый аккаунт должен иметь свой статус, своё время ответа и свою ошибку;
- сейчас при `tg:rescan-unread` общий замер всех аккаунтов может записываться каждому аккаунту как личный.

Причина в коде:

- `src/native/store/nativeStore.js` в `rescanUnread()` замеряет один общий `ms` вокруг `tg:rescan-unread`;
- затем `updateNativeHealthForAccounts(...)` применяет этот общий `ms` ко всем аккаунтам;
- в `main/native/telegramChats.js` функция `fetchAllUnreadUpdates()` уже считает время по каждому accountId в логах, но renderer пока не получает эти per-account данные как часть IPC-ответа.

Вывод:

```text
Native/API health должен быть строго per-account.
Общий замер операции нельзя показывать как личный замер каждого аккаунта.
```

### Что должно измениться в следующем этапе

⭐⭐⭐⭐⭐ Рекомендуемый вариант — исправить модель диагностики, а не только текст в UI.

1. WebView:

- `markHealthOk` не должен сам вычислять `lastMs` от старого `startedAt`, если явный `lastMs` не передан;
- сигналы `__CC_*` должны обновлять “вкладка живая” и время последней проверки, но не подменять время загрузки страницы;
- в списке `Подключения` показывать секунды только там, где это реальный замер загрузки/ответа.

2. Native/API:

- `tg:rescan-unread` должен возвращать статистику по каждому аккаунту;
- renderer должен обновлять `nativeConnectionHealth` отдельно для каждого `accountId`;
- если доступен только общий замер, его нельзя записывать в каждую учётку как личный.

3. UI:

- пользовательские статусы остаются русскими:

```text
pending → Ожидание проверки
ok      → Работает
slow    → Медленно отвечает
error   → Не отвечает / ошибка
Not connected → Не отвечает / ошибка
```

4. Тесты:

- добавить тест, что два Telegram API аккаунта не получают один и тот же общий `rescanUnread` замер как личный;
- добавить тест, что `markHealthOk` без явного `lastMs` не раздувает секунды от старого `startedAt`;
- проверить, что WebView `Работает` не показывает ложные сотни секунд.

### Варианты решения

#### ⭐⭐⭐⭐⭐ Вариант 1 — правильная per-account/per-operation модель

Суть:

- WebView хранит отдельно загрузку страницы и живой сигнал;
- Native/API хранит отдельные статусы по каждому аккаунту;
- IPC `tg:rescan-unread` возвращает per-account статистику.

Плюсы:

- исправляет корневую причину;
- подходит для будущих VK API, MAX API и других мессенджеров;
- пользователь видит, какой именно аккаунт или вкладка тормозит.

Минусы:

- нужно аккуратно менять main IPC и renderer store;
- нужны тесты на два Telegram API аккаунта.

Риск: 🟡 средний, если делать поэтапно.

#### ⭐⭐⭐⭐ Вариант 2 — быстрый безопасный фикс отображения

Суть:

- не показывать секунды WebView, если это не реальная загрузка;
- не писать общий `rescanUnread` замер каждому API аккаунту;
- если личного замера нет, показывать `Ожидание проверки` или только время последней проверки.

Плюсы:

- быстро убирает ложную картину;
- минимальный риск.

Минусы:

- точной per-account диагностики ещё не будет;
- проблему данных не решает полностью.

Риск: 🟢 низкий.

#### ⭐⭐⭐ Вариант 3 — расширенная диагностика по операциям

Суть:

- для каждого подключения показывать не одно число, а несколько операций:
  - WebView: загрузка страницы, последний живой сигнал, последняя ошибка;
  - Telegram API: получение чатов, unread rescan, состояние клиента, последняя ошибка.

Плюсы:

- максимум информации для расследования;
- удобно для сложных проблем сети.

Минусы:

- UI станет тяжелее;
- лучше делать после Варианта 1.

Риск: 🟡 средний.

Делать Вариант 1 поэтапно.
Сначала исправить модель данных и тесты, потом UI.

## Финальное решение без статуса "данные устарели"

Решение пользователя: отдельный статус `Данные устарели` не добавляем.

Оставляем только четыре пользовательских состояния:

```text
pending → Ожидание проверки
ok      → Работает
slow    → Медленно отвечает
error   → Не отвечает / ошибка
```

### Что показывать в списке "Подключения"

Для каждой WebView-вкладки и каждого Native/API аккаунта:

```text
точка | название | статус | последний ответ | последняя проверка
```

Пример:

```text
🟢 ВКонтакте             Работает             120 мс   13:37
🟡 Макс                  Медленно отвечает    3.8 сек  13:37
🟢 Telegram              Работает             95 мс    13:37
🔴 WhatsApp              Не отвечает / ошибка timeout  13:36

🟢 telegram · БНК         Работает            3.2 сек  13:37
🟢 telegram · Avtoliberty Работает            2.1 сек  13:37
```

Время ответа должно быть окрашено тем же цветом, что и точка статуса:

```text
ok    → зелёное время
slow  → жёлтое время
error → красное timeout/ошибка
```

### Что не показывать

Не показывать пользователю:

```text
Последняя загрузка страницы
```

Причина: пользователю нужен не технический lifecycle загрузки WebView, а понятный ответ:

```text
подключение сейчас отвечает или нет
```

### Как считать WebView

Для WebView больше нельзя использовать старое мутное `lastMs`, полученное из `startedAt` загрузки.

Первая реализация после исправления `startedAt` делала только локальный `executeJavaScript` probe:

```text
document.readyState
location.href
document.body
document.title
```

Ручная проверка показала проблему: при нажатии `Проверить все` все WebView-вкладки получили почти одинаковое время `47 мс`.

Вывод:

```text
локальный executeJavaScript probe измеряет скорость ответа Electron/WebView,
а не качество сети конкретного мессенджера.
```

Самопроверка показала ещё один риск: один только `/favicon.ico` может быть кэшем, CDN или иконкой, а не реальным маршрутом мессенджера.

Усиленный финальный замер WebView:

```text
1. Отправить внутрь WebView лёгкий JS probe.
2. Внутри вкладки сделать network fetch с cache-bust.
3. Сначала проверять текущий URL страницы без hash.
4. Если он не ответил — fallback на /favicon.ico.
5. Если он не ответил — fallback на /.
3. Замерить время fetch через performance.now() внутри WebView.
4. Вернуть это время как "Последний ответ".
```

Probe проверяет сетевую доступность текущего сайта вкладки:

```text
fetch(currentPageWithoutHash + ?cc_health=...)
fallback: fetch(origin + /favicon.ico?cc_health=...)
fallback: fetch(origin + /?cc_health=...)
cache: no-store
mode: no-cors
credentials: include
```

Это не перезагружает вкладку, не чистит cookies/session/partition и не меняет данные мессенджера.

### Как считать Native/API

Каждый API-аккаунт считается отдельно:

```text
telegram · БНК
telegram · Avtoliberty
```

У каждого аккаунта должны быть свои:

- статус;
- последний ответ;
- последняя проверка;
- последняя ошибка;
- операция, которая дала замер.

Общее время операции по всем аккаунтам нельзя записывать каждому аккаунту как личное время.

### Что реализовано в следующем этапе

- `markHealthOk` больше не вычисляет `lastMs` от старого `startedAt`, если явный `lastMs` не передан.
- При переходе в `Ожидание проверки` старое `lastMs` очищается, чтобы не было состояния `Ожидание проверки 1.5 сек`.
- Добавлен `src/utils/webviewHealthProbe.js`.
- Первая версия probe измеряла локальный ответ WebView и могла давать одинаковые значения по всем вкладкам.
- Исправлено: WebView probe теперь меряет `network-fetch` внутри вкладки и берёт `probeMs` из `performance.now()`.
- Самопроверка: один `/favicon.ico` признан слабым сигналом; probe усилен цепочкой `current-page → favicon → origin-root`.
- WebView lifecycle больше не подменяет "последний ответ" временем загрузки страницы.
- Кнопки проверки подключений запускают probe WebView без принудительного reload вкладок.
- `tg:rescan-unread` возвращает `accountStats` с личным `ms` по каждому accountId.
- Renderer обновляет `nativeConnectionHealth` по личным `accountStats`, а не по общему времени rescan.
- Фоновый `window.focus → rescanUnread` больше не перезаписывает health-замеры: он обновляет unread, но не меняет экран `Подключения`.
- `tg:get-cached-chats` больше не используется как сетевой health-замер, потому что это чтение локального кэша.
- В списке `Подключения` добавлена колонка "последняя проверка".
- Время ответа в списке окрашивается цветом статуса.

### Ручная проверка 8 мая 2026 — скачки pending/slow/ok у Native/API

Симптом:

```text
telegram · БНК → Ожидание проверки 1.5 сек
потом → Медленно отвечает 21 сек, операция tg:get-chats
через 2-3 секунды → Работает 2.1 сек, операция tg:rescan-unread
```

Причины:

1. `markHealthPending` сохранял старое `lastMs` из предыдущей проверки. Поэтому во время новой проверки строка показывала `Ожидание проверки`, но рядом оставалось старое время.
2. `window.focus` в `InboxMode` запускал `rescanUnread()`. После ручной проверки `Проверить все` операция `tg:get-chats` могла поставить `slow`, а фоновый `tg:rescan-unread` через несколько секунд перезаписывал эту же строку другим временем и статусом.

Фикс:

- `markHealthPending` теперь очищает `lastMs`.
- `InboxMode` вызывает `store.rescanUnread({ updateHealth: false })`: unread обновляется, но диагностика `Подключения` не перезаписывается фоновой операцией.

### Ручная проверка 8 мая 2026 — единая схема Native/API

Проблема после предыдущего фикса:

```text
Ручная кнопка "Проверить все" мерила tg:get-chats.
Фоновая проверка/фокус раньше могла мерить tg:rescan-unread.
Одна строка аккаунта показывала разные операции как будто это один и тот же ping.
```

Почему это было неправильно:

- `tg:get-chats` — тяжёлая загрузка списка чатов через `getDialogs`, кеш, emit и последующие фоновые работы.
- `tg:rescan-unread` — другая операция для непрочитанных.
- Время этих операций нельзя сравнивать как единый статус сети.
- Для панели `Подключения` должна быть одна схема проверки.

Что изменено:

- Добавлен IPC `tg:health-check`.
- `tg:health-check` делает лёгкий вызов `client.getMe()` для каждого Telegram API аккаунта.
- Он не грузит чаты, не читает кеш, не запускает аватарки и не пересчитывает unread.
- У каждого аккаунта возвращается личный `accountStats[].ms`.
- `NativeApp.refreshAll` больше не вызывает `loadCachedChats()` и `loadChats()` для панели `Подключения`.
- Кнопки `Проверить все` и `Обновить проблемные` теперь используют `store.checkConnection(accountId)`.
- `loadChats()` больше не записывает результат `tg:get-chats` в `nativeConnectionHealth`: загрузка чатов не является проверкой подключения.
- `rescanUnread()` по умолчанию не записывает результат `tg:rescan-unread` в `nativeConnectionHealth`: unread-синхронизация не является проверкой подключения.
- При появлении нового Native/API аккаунта `NativeApp` автоматически запускает первичный `store.checkConnection(accountId)` один раз. Без этого аккаунт мог зависнуть в `Ожидание проверки` до ручного нажатия кнопки.

Итоговое правило:

```text
Подключения / Native/API = только tg:health-check.
Загрузка чатов = только рабочая логика интерфейса.
Unread rescan = только обновление непрочитанных.
Эти операции больше не должны подменять друг друга в диагностике.
```

### Решение 8 мая 2026 — двухуровневый scheduler

Принятое решение: делать не постоянный ping каждой вкладки, а один общий scheduler для всего приложения.

Почему так:

- если проверять только при загрузке, ухудшение сети после старта будет видно поздно;
- если поставить `setInterval` на каждую вкладку, при 10+ вкладках начнутся дубли, гонки и лишняя нагрузка;
- один общий scheduler может контролировать очередь, параллельность и интервалы;
- WebView probe и Telegram API health-check остаются лёгкими и не смешиваются с рабочими операциями.

Правила scheduler:

```text
Один scheduler на всё приложение.
Не больше 2 WebView проверок одновременно.
Не больше 1 API проверки одновременно.
Активное подключение: раз в 30 секунд.
Неактивные подключения: раз в 3 минуты.
Проблемные подключения: раз в 60 секунд.
Если окно не в фокусе: раз в 5 минут.
Если проверка уже идёт: пропустить.
Если WebView сейчас грузится: не мешать, дождаться окончания загрузки.
Ручная кнопка идёт вне очереди, но тоже без дублей.
```

Ожидаемая нагрузка при 10 WebView вкладках:

```text
1 активная вкладка: примерно 2 проверки в минуту.
9 неактивных вкладок: примерно 3 проверки в минуту суммарно.
Итого около 5 лёгких WebView probe в минуту.
API аккаунты проверяются отдельно, максимум 1 одновременно.
```

Это нормально по CPU/памяти:

- scheduler хранит только маленькое состояние: `id`, `lastCheckedAt`, `state`, `inFlight`, `nextCheckAt`;
- WebView probe делает лёгкий `fetch` внутри вкладки и не перезагружает страницу;
- API probe делает `tg:health-check` (`client.getMe()`), а не загрузку чатов;
- главная память всё равно уходит на сами WebView, а не на диагностику.

Что важно не нарушить:

- не запускать проверки всех вкладок одновременно;
- не проверять WebView во время загрузки страницы;
- не превращать `tg:get-chats`, unread rescan или аватарки в health-check;
- не запускать второй health-check, если первый ещё выполняется.

### Уточнение 8 мая 2026 — активный Native/API аккаунт

После повторной проверки решения найден слабый момент: если считать весь `ЦентрЧатов` активным, то все Telegram API аккаунты получают интервал активной проверки 30 секунд. Для 2 аккаунтов это почти незаметно, но для 30 API-подключений это уже лишняя очередь и ненужные запросы.

Итоговое правило:

```text
Активный WebView = открытая верхняя вкладка.
Активный Native/API = аккаунт открытого чата.
Если чат не открыт, но выбран фильтр аккаунта — активен выбранный аккаунт.
Если выбран "Все" и чат не открыт — активного API аккаунта нет.
Остальные API аккаунты считаются неактивными и проверяются реже.
Проблемные аккаунты всё равно проверяются чаще: раз в 60 секунд.
```

Почему это правильнее:

- пользователь сейчас работает с конкретным чатом/аккаунтом, а не со всеми API аккаунтами сразу;
- при 10, 30 и большем количестве API-подключений частая проверка не размножается на все аккаунты;
- лимит `1 API-проверка одновременно` сохраняется;
- новые API аккаунты автоматически попадают в список через `NativeApp` snapshot и получают первую проверку один раз;
- схема не трогает Telegram-сессии, загрузку чатов, unread, аватарки и данные пользователя.

### Уточнение 8 мая 2026 — обратный отсчёт до проверки

В панели `Подключения` добавлен показатель `Следующая проверка`.

Это только UI-расчёт:

```text
lastCheckedAt + интервал scheduler - текущее время
```

Он не запускает сеть, не делает IPC и не проверяет соединение. Пока панель открыта, один общий UI-таймер обновляет надпись раз в 5 секунд. Это не создаёт заметной нагрузки даже при большом числе подключений.

### Самокритика решения

Что выбрано удачно:

- health-check отделён от рабочих операций: `tg:health-check` не грузит чаты, а WebView probe не перезагружает вкладку;
- один scheduler на всё приложение лучше, чем отдельные таймеры на каждую вкладку;
- ограничения параллельности защищают приложение от всплесков нагрузки;
- активный API аккаунт теперь определяется по реальному контексту пользователя.

Минусы, которые остаются осознанно:

- `tg:health-check` проверяет доступность API аккаунта через лёгкий `getMe()`, но не гарантирует, что тяжёлая загрузка чатов будет быстрой;
- WebView probe показывает сетевой ответ сайта, но не заменяет полноценную диагностику самого сайта, если сервис внутри завис;
- при 30 API аккаунтах полный круг проверки неактивных аккаунтов может занимать несколько минут. Это сделано специально, чтобы диагностика не мешала работе.

Что не надо добавлять сейчас:

- отдельный постоянный ping на каждый аккаунт;
- проверку API через `tg:get-chats`;
- проверку WebView через reload;
- состояние "данные устарели" — пользователь от него отказался.
