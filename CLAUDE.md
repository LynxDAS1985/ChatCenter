## 🧠 БАЗОВЫЕ ПРИНЦИПЫ РАБОТЫ

### 1. THINK BEFORE CODING
- Перед правками коротко напиши, как ты понял задачу.
- Назови допущения, если они есть.
- Если задача двусмысленная и ошибка может увести не туда — задай вопрос.

### 2. SIMPLICITY FIRST
- Делай минимально достаточное решение.
- Не добавляй код «на будущее».
- Не строй абстракции для одного случая.
- Не рефактори соседний код, если это не нужно для текущей задачи.

- Трогай только связанные с задачей файлы и строки.
- Не меняй несвязанный код, форматирование, комментарии, имена, структуру проекта без необходимости.
- Если видишь побочную проблему — сначала скажи о ней, но **не лезь чинить без запроса**.

### 4. GOAL-DRIVEN EXECUTION
- Перед началом сформулируй критерий готовности:
  **«Задача готова, когда: …»**
- После правок обязательно проверь именно этот критерий.
- Не считай задачу завершённой просто потому, что код написан.

## ⚡ ПРАВИЛА ВЫПОЛНЕНИЯ

### 1. ПАРАЛЛЕЛЬНОСТЬ — делай максимум за один ход
- Читай все нужные файлы **ОДНОВРЕМЕННО** (несколько `Read` в одном сообщении)
- Делай все независимые правки **ОДНОВРЕМЕННО** (несколько `Edit` в одном сообщении)
- Запускай Agent'ов **ПАРАЛЛЕЛЬНО**, когда задач несколько и они не зависят друг от друга
- **НЕ** разбивай одну задачу на 10 сообщений — для средней задачи старайся уложиться в 3–4 хода
- Параллелить можно **только если НЕТ зависимости по данным**. Если шаг Б зависит от шага А — делай последовательно.

### 2. ЧТО НЕЛЬЗЯ ПАРАЛЛЕЛИТЬ
- `apply_patch`
- git-команды
- `npm install` / `npm ci`
- build
- e2e
- команды, которые пишут в одни и те же файлы, кэш или каталоги

### 3. PIPELINE ВЫПОЛНЕНИЯ

**ПРОСТАЯ ЗАДАЧА** (1 файл, 1 локальная правка):
за 1 ход — прочитал → изменил → проверил → отчитался

**СРЕДНЯЯ ЗАДАЧА** (2–5 файлов):
- **ХОД 1:** Прочитай `CLAUDE.md` + `.memory-bank` по теме + нужный код (параллельно). Сформулируй критерий готовности.
- **ХОД 2:** Сделай **ВСЕ** нужные правки кода. Если код менялся — обнови версию во всех 4 местах.
- **ХОД 3:** Обнови `.memory-bank`. Запусти проверки. Сделай commit + push — **но только если пользователь явно попросил** в текущей задаче.
- **ХОД 4:** Итог + «Что проверено / Что не удалось» + до 5 советов (см. ниже) + таблица «Что сделано в этой сессии».

**КРУПНАЯ ЗАДАЧА** (>5 файлов / новая фича / архитектурный риск):
сначала краткий план с количеством ХОДов, потом выполнение.

**После каждого хода писать**: `ХОД X/N: ✅ [что сделано]`

## 🔄 КОНФЛИКТ: ПАМЯТЬ vs КОД

Если `.memory-bank/*` или auto-memory (`.claude/memory/*`) говорят одно, а реальный код — другое → **доверяй коду**.

**Почему**: память живёт долго, код меняется. Запись в памяти может устареть — например, была `scripts/dev.js`, стала `scripts/dev.cjs`; упоминается функция, которой уже нет; ссылается на файл, который переименован.

**При расхождении**:
1. Использовать текущее состояние кода
2. Обновить соответствующую запись в Memory Bank (или auto-memory)
3. Если не уверен — **спросить пользователя**

**Аналогично для размеров файлов**: если в CLAUDE.md указан «текущий размер 556 строк», а реально 620 — сначала проверь реальность через `wc -l`, потом обновляй доку.

## 🎓 Контекст проекта

**Название**: ЦентрЧатов (рус.) / ChatCenter (eng.)

**Что это**: Electron-приложение — единое окно для работы с несколькими мессенджерами одновременно, с встроенным ИИ-помощником для быстрых ответов, шаблонов и автоответа.

**Целевая аудитория**: Операторы и менеджеры, работающие с клиентами через несколько мессенджеров (Telegram, WhatsApp, VK, Viber, MAX и др.).

## 🏗️ Архитектурные принципы

**Технологии**:
- **Electron** — оболочка приложения
- **WebView** (`<webview>` тег) — изолированный рендер веб-версий мессенджеров
- **IPC** (`ipcMain / ipcRenderer`) — единственный канал связи `main ↔ renderer`
- **preload.js** — мост между `main` и `renderer` (`contextBridge`)

**Критически важно**:
- Всё через IPC — никакого `require('electron')` в renderer-процессах
- Инъекция скриптов в WebView только через `preload` WebView или `executeJavaScript`
- Каждый мессенджер — отдельный изолированный `<webview>` с собственной сессией/partition
- Перехват сообщений — через `MutationObserver` внутри WebView, результат передаётся через IPC
- ИИ-запросы только из main-процесса (API-ключи не в renderer)

## 📏 Правило превышения лимита строк

Если файл превышает лимит строк — **НЕ резать комментарии**. Комментарии должны оставаться всегда, они объясняют логику и предупреждают о ловушках.

## 🚫 Лимиты размера файлов КОДА (src/, main/)

Проект защищён от бесконтрольного роста файлов. Тест `fileSizeLimits.test.cjs` автоматически проверяет лимиты.

### Лимиты на файлы (v0.85.0+)

| Файл | Лимит | Если превышен |
|------|-------|--------------|
| Любой `.jsx` компонент | **600 строк** | Разбить на подкомпоненты или вынести логику в utils |
| Любой `.js` утилита | **300 строк** | Разбить на несколько файлов по функциям |
| Любой React hook | **130 строк** | Разбить на несколько хуков |
| `src/App.jsx` | **600 строк** | Вынести в `src/hooks/` или `src/components/` |
| `main/main.js` | **600 строк** | Вынести в `main/handlers/` или `main/utils/` |
| `monitor.preload.cjs` | **600 строк** | Вынести в `main/preloads/utils/` |
| `webviewSetup.js` | **600 строк** | Вынести в `src/utils/` |
| `AISidebar.jsx` | **600 строк** | Вынести в `src/utils/ai*.js` |
| `dockPinHandlers.js` | **600 строк** | Вынести в `main/handlers/dockPinUtils.js` |

### Лимиты на функции

| Правило | Лимит | Если превышен |
|---------|-------|--------------|
| Одна функция | **100 строк** | Разбить на подфункции |
| Inline `executeJavaScript` | **50 строк** | Вынести скрипт в `messengerConfigs.js` |
| React `useEffect` | **30 строк** | Вынести логику в отдельную функцию |

### Правила добавления нового кода

1. Новая бизнес-логика → `src/utils/имяМодуля.js` + тест в `src/__tests__/`
2. Новый UI-блок → `src/components/ИмяКомпонента.jsx` (если > 50 строк)
3. Новый IPC handler → добавить в `main/main.js` + тест в `ipcChannels.test.js`
4. Новый спам-паттерн → `shared/spamPatterns.json` (единый источник)
5. Новый мессенджер → конфиг в `src/utils/messengerConfigs.js`

### Как проверить (без запуска приложения)

```bash
npm run lint
npm run test:vitest
```

| 2 | `package-lock.json` | корневой `"version"` + `packages[""].version` (две записи) |
| 3 | `CLAUDE.md` | `**Текущая версия**` и `**Версия проекта**` (в начале и в подвале) |
| 4 | `.memory-bank/features.md` | Версия в шапке + запись в Changelog |

### Правила версионирования
- **Patch** (0.1.X → 0.1.X+1): багфиксы, мелкие правки, рефакторинг памяти
- **Minor** (0.X.0): новые функции, новый мессенджер, изменения UI
- **Major** (X.0.0): только по согласованию с пользователем

### Когда версию НЕ повышать
Если код **не менялся**, а был только анализ / совет / текст для обсуждения — версию **не** повышать.

### Чеклист перед сборкой
1. ✅ `package.json` — версия
2. ✅ `package-lock.json` — две `"version"`-записи
3. ✅ `CLAUDE.md` — версия и дата в двух местах
4. ✅ `.memory-bank/features.md` — версия + запись в Changelog
5. ✅ Если есть новые файлы — добавлены в `package.json → build.files`

### 📏 Лимиты размера файлов ПАМЯТИ (.memory-bank/)

| Файл | Максимум | Если превышен |
|---|---|---|
| Любой `.md` в корне `.memory-bank/` | **100 КБ** | Разбить на подпапку + индекс |
| Файл в `mistakes/` | **200 КБ** | Разбить по подтемам |
| `common-mistakes.md` (индекс) | **10 КБ** | Сократить описания тем до одной строки |

**Почему лимиты важны**: инструмент `Read` имеет предел 256 КБ. Файл больше — не читается. 100+ КБ съедает огромную долю контекста, нужного для задачи.

**Автоматическая защита**:
- **Тест лимитов**: `node src/__tests__/memoryBankSizeLimits.test.cjs` — падает если файл перерос
- **Тест dangling-refs**: `node src/__tests__/featuresReferences.test.cjs` — падает если в последних 10 версиях `features.md` ссылка указывает на несуществующий файл
- **Pre-commit hook**: `scripts/hooks/pre-commit` вызывает `check-memory.sh` если в коммите есть `.memory-bank/*` или `CLAUDE.md` → коммит блокируется при проблемах
- **Скрипт диагностики**: `bash scripts/check-memory.sh` (или `npm run check-memory`) — размеры, сверка версий, сломанные ссылки
- **Регенерация структуры**: `bash scripts/regen-claude-structure.sh` (или `npm run regen-claude-structure`) — автоматически обновляет таблицу «Структура памяти» между маркерами `<!-- STRUCTURE-AUTO-START/END -->`

## 🔧 GIT-ПРАВИЛА

1. **Перед commit всегда** смотри `git status`
2. Проверяй, нет ли незакоммиченных чужих изменений
3. Добавляй файлы **ТОЛЬКО ПО ИМЕНАМ**: `git add file1 file2 file3`
4. **НЕ** использовать `git add -A` / `git add .`
5. `git commit --no-verify` — **ЗАПРЕЩЁН** без явного разрешения пользователя
6. Один логический запрос пользователя = один логический commit
7. **Commit + Push делать только если пользователь ЯВНО попросил** это в текущей задаче
8. Формулировка «кажется, тут нужен push» — **не основание** для push

### Если делаешь add + commit + push одной командой Bash
- Это должна быть **одна последовательная команда** (`&&`)
- **Не** параллельные git-вызовы
- **Только после** успешных проверок
- **Только если** пользователь просил commit/push

## 💬 ФОРМАТ ОТВЕТА

### 🚨 ГЛАВНОЕ ЖЁСТКОЕ ПРАВИЛО — простой язык

Пользователь **НЕ программист**. Все ответы, советы, объяснения — **простыми словами**.

**ЗАПРЕЩЕНО** без крайней необходимости:
- Технические термины (`refactoring`, `context isolation`, `throttle`, `race condition`, `deadlock`, `dependency`)
- Слова вроде «синхронизация», «интерпретация», «агрегация», «абстракция»
- Профессиональный жаргон, англицизмы, код-термины в объяснениях

**РАЗРЕШЕНО**:
- Простые слова: «одновременно», «вовремя», «вместе», «по очереди»
- Объяснение через аналогии («это как…»)
- Имена файлов и команд — да, они не заменяются (например `package.json`)

**Тест на понятность**: если объяснение нельзя пересказать бабушке — переписать проще.

**Пример**:
- ❌ «Параллелизм — делай независимые операции одновременно для оптимизации throughput»
- ✅ «Одновременно — делай несколько дел сразу, когда они не мешают друг другу»

### Общие правила

- Все мысли, действия и итог пиши **НА РУССКОМ**
- Коротко, но достаточно полно
- Информации давай столько же или больше, **но за меньше ходов**
- Не растягивай очевидные действия на лишние сообщения

### Для разных типов задач

**Советательная / аналитическая задача** — в ответе отдельно разделяй:
1. Что уже проверено
2. Что предлагается
3. Что ещё не применялось к файлам

**Смешанная задача** (часть совет, часть код):
1. Что изменено в коде
2. Что только предлагается
3. Что пользователь должен проверить сам

### «До 5 советов по улучшению» — только для содержательных задач

Дай **ДО 5 вариантов улучшения ТОЛЬКО если** задача была содержательной:
- Новая фича
- Рефакторинг
- Крупный фикс
- Архитектурное решение

**Для тривиальных правок** (typo, одна строка, переименование переменной) советы **не давать** — это выглядит как избыток.

Если всё же даёшь советы — для каждого обязательно укажи:
- Что это даст
- Зачем нужно
- Простой пример / визуальный пример (если возможно)
- Плюсы / минусы
- Что будет, если не сделать
- Какие проблемы могут быть, если сделать

Пиши **без сложных слов**, понятно человеку без технической подготовки. См. «🚨 Главное жёсткое правило» в начале этой секции.

### 🎨 Визуальное оформление советов

Советы **должны выглядеть**, а не «просто быть написаны». 5 приёмов:

#### Приём 1 — 📊 Таблица «До / После»
Когда совет про изменение текста, числа, строки, версии — две колонки:

```
| ❌ Сейчас          | ✅ Станет              |
|---|---|
| 76 строк старого   | 138 строк актуальных   |
```

Применять: правки файлов, замены значений, переименования.

#### Приём 2 — 🟢🟡🔴 Иконки-светофор вместо слов «плюсы/минусы/риски»
Вместо длинных заголовков — одна иконка + одна строка:

```
🟢 Что даст        → быстрый ответ одной фразой
🟡 Цена            → 5 минут работы, 1 коммит
🔴 Если не сделать → что сломается
⚡ Срочность       → низкая / средняя / высокая
```

Применять: всегда когда есть баланс «стоит ли делать».

#### Приём 3 — ⭐ Шкала приоритета (1–5 звёзд)
Вместо слов «важно / очень важно / критично»:

```
⭐⭐⭐⭐⭐ — блокер, сделать первым
⭐⭐⭐   — полезно, следующее
⭐      — косметика, можно отложить
```

Применять: когда в ответе **несколько** советов и надо понять что делать первым.

#### Приём 4 — 🖼 ASCII-мокап для структуры и архитектуры
Когда совет про структуру папок, компонентов или UI — нарисуй:

```
Было:
.memory-bank/
├── common-mistakes.md  [294 КБ] ❌ не читается
└── features.md         [445 КБ] ❌ не читается

Стало:
.memory-bank/
├── common-mistakes.md  [5 КБ]   ✅ индекс
├── mistakes/           [темы]    ✅
└── archive/            [история] ✅
```

Применять: структура, архитектура, UI-макеты.

#### Приём 5 — 🚦 Одна строка-светофор для мелких советов
Если совет мелкий — не расписывай 6 пунктов. Одна строка:

```
🟢 Убрать цифру «76 КБ» из CLAUDE.md → стареет быстро (5 секунд)
🟡 Разбить webview-stack-grouping.md → можно отложить (средний приоритет)
🔴 Удалить архив → НЕ делай, потеряем учебные примеры
```

Применять: когда 5 советов подряд — чтобы не было стены текста.

### Какой приём для какого случая

| Тип совета | Приём |
|---|---|
| Правка текста / числа / строки | 📊 До/После (Приём 1) |
| Список из 3–5 советов с разным приоритетом | ⭐ Шкала (3) + 🚦 светофор (5) |
| Структура папок, архитектура, UI | 🖼 ASCII-мокап (4) |
| Плюсы/минусы/риски в одном совете | 🎯 🟢🟡🔴 иконки (2) |
| Очень мелкая правка | 🚦 Одна строка (5) |

**Главное**: совет должен **бросаться в глаза**, а не теряться в тексте. Если ты написал 3 страницы и главное на 2-й — это плохой формат.

## ⚙️ Конфигурация `.claude/`

В корне проекта есть папка `.claude/` с настройками Claude Code:

### `.claude/settings.json`
Project-specific разрешения. Сейчас разрешены без подтверждения:
- `Bash(*)`, `Read(*)`, `Write(*)`, `Edit(*)`
- `Glob(*)`, `Grep(*)` — поиск по коду
- `WebFetch(*)`, `WebSearch(*)` — интернет-запросы
- `Agent(*)` — подагенты для независимых задач
- `TodoWrite(*)`, `NotebookEdit(*)`

**Почему это важно**: агент не спрашивает подтверждения перед каждой командой. Это ускоряет работу, но **не отменяет критические запреты** (`npm start`, `git reset`, `npm test` и т.д.) — эти команды агент не должен запускать даже если `Bash(*)` разрешён.

### `.claude/skills/` и `.claude/commands/`
Сейчас **пусто**. Если появятся skills/slash-команды — **обязательно добавить их в CLAUDE.md** (см. раздел «MEMORY BANK — обязательные действия»).

### `.claude/memory/`
Auto-memory — постоянная память Claude между сессиями. При расхождении с кодом — **доверять коду**, память обновлять (см. раздел «Конфликт: память vs код»).
